## Anti-Patterns Verdict

Does NOT read AI-generated. Enforced opinionated system: one palette, zero radius, tabular-nums, hairline borders, no gradient text, no hype copy. Detector: 11 findings, mostly advisory; 2 real token leaks (#b7791f line 441; theme-color meta #E8612A ≠ accent #FF4A1C). ## Priority Issues

**[P0] Rest timer & session don't survive lock screen or reload.** No Wake Lock, no vibration, no notification; restState/liveTimerStart unpersisted; iOS suspends setInterval when locked. The rest-timer-in-your-pocket is THE load-bearing elite feature. Fix: persist endAt timestamps + recompute on load; wakeLock.request('screen') during sessions; navigator.vibrate + optional Notification on rest end; ±15s adjust next to Skip.

**[P1] Corrupted state = silent permanent history wipe.** load() catch{} falls to freshState; next save() overwrites the recoverable blob. Fix (~5 lines): quarantine the raw blob to a second key + toast.

**[P1] saveSoon() 400ms debounce, no pagehide flush.** Weight/reps/RPE edits lost on iOS swipe-kill within the window. Fix: addEventListener("pagehide", save) + visibilitychange flush.

**[P1] No per-exercise history at the point of need.** History sheet = top-8 by frequency, axis-less 70x24 sparklines, no PRs, no dates. Elite users check a lift's trajectory before picking a weight. Data already exists (buildHistoryData). Fix: tap exercise name → dated est-1RM series + rep PRs; flag PRs in the Complete receipt.

**[P1] One Accent Rule broken by its own screens.** Home: accent planBtn + orange CTA + accent NEXT tag simultaneously. Active: orange progress fill + LOG bar + now-chip + orange DONE stamp. Fix: demote planBtn once a program exists; DONE stamp → ready-green; NEXT → neutral-strong.

**[P1] Contrast + micro-type fail WCAG.** --faint #565B62 = 2.47–2.86:1 on its backgrounds (needs 4.5); light theme worse (2.37). 39 sub-12px font sizes incl. two 9px. Fix: bump --faint toward #6E747D+; floor labels at 10–11px.

**[P2] Set-row crowding at 375px.** KG "+" and REPS "−" steppers collide; REPS "+" partially under the done circle; 7 controls per row vs Strong's 3. Fix: tighten row grid or demote steppers behind a row tap.

**[P2] Unit handling cosmetic-broken.** "Kg" column header hardcoded (setColHead); verify kg↔lb flip converts stored weights vs relabels. Kills "trust the numbers" for lb users.

**[P2] Import-borne stored XSS.** migrateLogEntry passes migrated el.sets through unvalidated; crafted backup actual_weight string reaches raw attribute interpolation. Fix: numeric-coerce in migrateLogEntry/readLog; add CSP meta.

**[P2] SW Lie-Fi stall.** Network-first navigation has no timeout race; stalled connection blocks launch until browser timeout. Fix: Promise.race with ~3s cache fallback. Also: assets cache-first-forever within a version — bump CACHE_VERSION every deploy or suffix asset URLs.

**[P2] No exercise media.** Text-only cues. Fix consistent with offline: ~20 inline SVG position diagrams per movement_pattern, not per exercise.

**[P2] Multi-tab clobbering.** Last-writer-wins whole blob. Fix: savedAt stamp + reload-and-warn on mismatch.

## What's Working

1. refreshLogBar: sticky LOG dispatches the real set-row control — one code path, honest muted advance state. Elite-tier.
2. Systemic undo-toast pattern with deep snapshots on every destructive action — Strong/Hevy don't do this consistently.
3. buildHonestRead + platesText + warm-up ramps: the why-line pipeline offline in one file is what Fitbod charges for.
4. Engineering discipline: esc() near-perfect, idempotent migrations, no timer leaks, surgical DOM patches, clean z-index/motion/reduced-motion.

## Persona Red Flags

**Alex (Power User / serious lifter):** no per-lift history or PRs at point of need; no quick-repeat of last session; no exercise reorder; rest timer dies in pocket. Would churn back to Strong in a week.
**Jordan (First-Timer / future user):** RPE and est. 1RM never explained; icon-only header buttons; text-only form cues for movements they've never seen; 9px labels in gym lighting.
**Sam (Data-Trusting Skeptic — brand persona):** "Kg" header in lb mode; "Log set 1 — 0 kg" advertising a zero-weight log; app knows "add 2.5kg next time" but still makes the user choose the weight.

## Questions to Consider

1. If the app trusts its own numbers, why does the user still choose the weight? Pre-fill the earned increment; make declining the edit.
2. Is the recovery-ring donut earning the top of Home, or would its one honest sentence serve the instrument brief better?
3. When does honesty need a next move? "Pulled back to 87kg — logged as-is" without "so next session holds 85" reports the stall but not the corrective input.
